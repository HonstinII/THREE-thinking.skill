# THREE-thinking

三者搭配形成 **"拆解→追本→精简"** 的闭环，让思考更深度、决策更精准、行动更高效，避免 "想得多、做得杂、抓不住重点" 的问题。

---

## 这是什么？

**THREE-thinking** 是一套基于三大经典思维方法的 AI Skill 集合，帮助你在面对复杂问题时，从"凭感觉决策"升级为"系统化思考"。

三大思维工具：
- **苏格拉底式提问** —— 连续追问，拆解问题本质
- **第一性原理** —— 抛开表象，回归事物底层逻辑  
- **奥卡姆剃刀** —— 如无必要，勿增实体，精简方案

---

## 解决什么问题？

### ❌ 不使用这些工具时，常见的思考陷阱：

| 陷阱 | 表现 | 后果 |
|------|------|------|
| **经验主义** | "以前这么做成功了，现在继续" | 忽视环境变化，重复过时方案 |
| **从众思维** | "别人都在做，我们也要做" | 盲目跟风，失去差异化优势 |
| **表面归因** | 看到症状就下结论 | 治标不治本，问题反复出现 |
| **过度复杂** | 方案越堆越多，功能越加越杂 | 资源分散，核心目标模糊 |
| **信息过载** | 收集大量资料却无法决策 | 分析瘫痪，行动拖延 |

### ✅ 使用后的改变：

```
Before: "我觉得应该加个推荐算法" → 盲目开发 → 用户不买账
After:  "用户真正的痛点是什么？" → 追问本质 → 发现是内容质量 → 针对性解决
```

## 包含的 Skills

### 1. three-thinking-tools（通用版）

**语言**: 中文  
**定位**: 通用思维方法论，适用于任何需要深度思考的场景

**解决的核心问题**:
- 面对复杂决策时无从下手
- 被表面现象迷惑，找不到根本原因
- 方案越做越复杂，抓不住重点

**典型使用场景**:
- 🎯 **产品决策**: 新功能要不要做？优先级如何排？
- 📚 **学习规划**: 如何高效掌握一门新技能？
- 🚀 **创业方向**: 要不要进入某个赛道？差异化在哪？
- 💼 **职业选择**: 换工作还是留任？长期目标是什么？
- 🤝 **团队协作**: 效率低下，问题到底出在哪？

**核心口诀**: 先问为什么 → 再找本质因 → 最后做减法

---

### 2. design-thinking（软件设计版）

**语言**: 英文  
**定位**: 专为软件开发设计，与 [superpowers](https://github.com/obra/superpowers) 工作流集成

**解决的核心问题**:
- 需求模糊， stakeholders 意见不一致
- 想抄竞品，但不知道是否适合自己
- 功能越加越多，违背 YAGNI 原则

**典型使用场景**:
- 💻 **功能设计**: 用户说"想要一个 dashboard"，但真实需求是什么？
- 🏗️ **架构决策**: 要不要引入微服务？本质是为了解决什么问题？
- 📝 **需求澄清**: 产品经理和开发对需求理解不一致
- 🎨 **方案选择**: 多个技术方案，如何选出最简单的那个？

**与 superpowers 集成**:
```
用户提出需求 → design-thinking 澄清 → brainstorming 设计 → writing-plans 实现
```

## 两个 Skill 的区别

| 维度 | three-thinking-tools | design-thinking |
|------|---------------------|-----------------|
| **语言** | 中文 | 英文 |
| **适用范围** | 通用（产品/学习/创业/生活） | 专用（软件设计/开发） |
| **目标用户** | 产品经理、创业者、学习者 | 软件工程师、技术负责人 |
| **集成场景** | 独立使用 | 与 superpowers 工作流结合 |
| **输出重点** | 思考过程 + 决策框架 | 澄清后的需求 + 精简的设计 |
| **示例问题** | "要不要创业？" | "这个 API 要不要加缓存？" |

### 如何选择？

```
你在做软件设计/开发？
    ├── 是 → 使用 design-thinking（英文）
    │         └── 与 superpowers 配合效果更佳
    │
    └── 否 → 使用 three-thinking-tools（中文）
              └── 产品决策、学习规划、创业思考等通用场景
```

## 不使用 vs 使用对比

### 场景：团队效率低下

| 阶段 | ❌ 不使用思维工具 | ✅ 使用思维工具 |
|------|------------------|----------------|
| **问题识别** | "大家不够努力，需要加班" | "效率低的具体表现是什么？哪个环节最耗时？" |
| **原因分析** | "员工态度问题" | "流程设计是否违背了分工协作的底层逻辑？" |
| **解决方案** | 加人 + 加班 + 加激励 | 明确分工 + 简化审批 + 核心节点监督 |
| **结果** | 成本增加，问题依旧 | 流程顺畅，效率提升 |

### 场景：产品功能规划

| 阶段 | ❌ 不使用思维工具 | ✅ 使用思维工具 |
|------|------------------|----------------|
| **需求来源** | "竞品有这个功能" | "用户的核心痛点是什么？" |
| **功能设计** | 复制竞品功能 | 从本质需求推导最小可行方案 |
| **优先级** | 功能越多越好 | 只保留解决核心问题的功能 |
| **结果** | 产品臃肿，用户迷失 | 简洁有力，用户满意 |

## 核心哲学

> **好的思考是一种可以系统化的技能。**

这套技能基于三个信念：

1. **深度理解源于质疑** —— 不要接受表面答案，持续追问为什么
2. **本质比表象更重要** —— 抛开类比和经验，回归第一性原理
3. **简单是终极的复杂** —— 如无必要，勿增实体，精简即美
